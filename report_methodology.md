# AI 輔助開發的分層品質保證流程

## 動機：在 AI 加速下，如何守住程式碼品質

本專案在開發過程中使用 Claude Code 作為 AI 輔助開發工具。然而，直接讓 AI 一次生成全部程式碼，在這個專案中是有風險的——本專案以 valgrind 零記憶體洩漏作為硬性評分門檻，而影像資料採用 `int**`（灰階）與 `int***`（RGB）的多層動態配置，每一次 `new[]` 都必須對應正確的巢狀 `delete[]`。在這種對記憶體安全要求極高的情境下，「能編譯、能執行」遠遠不夠，任何一處漏接 `delete`、釋放順序錯誤、或淺拷貝造成的 double-free，都會在 valgrind 下現形而失分。

因此，本專案並未把 AI 當成「程式碼產生器」直接採用其輸出，而是設計了一套**分層的品質保證流程**：由 plan mode 負責架構對齊、由自訂 subagent 負責自動化品質把關、再由人工負責逐塊驗收。這套流程的核心理念是——**利用 AI 的生產力，同時透過流程設計守住程式碼品質這條底線**。以下說明三個層次的設計。

## 第一層：Plan Mode——架構先行

每一個開發階段（Step）開始時，先進入 Claude Code 的 plan mode，要求其在閱讀規格與既有程式碼後，**先產出一份完整的實作計畫，而非直接動手寫程式**。計畫內容包含：要新增或修改哪些檔案、類別介面設計、演算法步驟、以及——最關鍵的——一份明確的「記憶體安全清單」，逐項列出每個 `new[]` 對應的釋放時機、每條失敗路徑（early return）的清理責任、以及物件所有權的歸屬。

這份計畫會經過人工審查後才核准執行。這個「架構先行」的步驟之所以重要，是因為記憶體管理的正確性往往取決於**整體設計**而非單行程式碼——例如「鄰域濾鏡必須配置新緩衝區再透過 `replace_buffer` 接管，而非原地修改」這類決策，若不在計畫階段就釐清，等到實作時才發現，重工成本極高。計畫先行讓設計層級的記憶體風險在寫任何一行程式碼之前就被討論清楚。

## 第二層：自訂 Subagent——自動化品質把關

本專案在 `.claude/agents/` 下設計了兩個職責單一、互補的自訂 subagent，將品質檢查自動化。兩者的設計都遵循「**單一職責**」與「**唯讀、不修改程式碼**」的原則——它們只負責「發現問題並回報」，修正的決策權保留在主開發流程，避免 AI 在未經審查下自行改動程式碼。

### mem-checker：記憶體安全把關

`mem-checker` 負責自動化的 valgrind 驗證，其設計重點如下：

- **工具配置**：Bash、Read、Grep。需要 Bash 以在容器內執行編譯與 valgrind。
- **模型選擇**：使用較輕量的 haiku 模型。因為這個 agent 的工作偏向機械性的執行與輸出過濾，不需要複雜的語意判斷，輕量模型即可勝任且回應更快。
- **工作流程**：先執行 `make -j`，若編譯失敗則回報編譯錯誤並停止；編譯成功則執行 `make check`（valgrind --leak-check=full）。
- **領域知識內建**：此 agent 的指令中明確寫入了本專案特有的記憶體結構知識——「pixels 是 `int**`（灰階）與 `int***`（RGB），釋放需要巢狀迴圈而非單一 delete」。將專案的領域知識教給 agent，使其檢查更精準、能直接指出缺少對應 `delete[]` 的具體 `new[]` 位置。
- **雜訊過濾**：明確忽略第三方函式庫（CImg / libjpeg）所產生的 "still reachable" 訊息，只聚焦於專案自身程式碼的洩漏與錯誤，讓回報的清單聚焦、可行動。

### oop-reviewer：OOP 設計把關

`oop-reviewer` 負責對照評分標準審查物件導向設計，其設計重點如下：

- **工具配置**：Read、Grep、Glob。純粹靜態閱讀程式碼，不需要執行，因此不配置 Bash。
- **模型選擇**：使用能力較強的 sonnet 模型。與 mem-checker 不同，OOP 設計審查需要對程式碼進行語意層面的判斷（例如 friend 用得是否得當、是否破壞封裝、繼承關係是否正確），複雜度較高，因此配置較強的模型。**依任務複雜度選擇對應的模型，本身就是一個有意識的設計決策。**
- **評分標準編碼化**：此 agent 最核心的設計，是將專案的 OOP 評分標準直接編碼成審查準則，包括：基底類別 `Image` 的 `width`/`height`/`data_loader` 須為 protected、四個介面函式須為純虛擬（`= 0`）、解構子須為 virtual；衍生類別須為 public 繼承、像素陣列須為 private、四個虛擬函式皆須 override；`Data_Loader` 須以 static 或 Singleton 共享而非每個物件各建一份；遵守 rule of three/five（擁有 `new[]` 記憶體的類別須有解構子，並明確定義或刪除拷貝建構子與拷貝賦值，以防 double-free 與淺拷貝）；以及透過 `Image*` 指標的動態繫結是否會發生 object slicing。**等於把評分標準變成了一個可重複執行的自動化審查器。**
- **分級回報**：審查結果依優先序分為 Critical（評分標準違反或記憶體安全錯誤，必須修正）、Warning（可能的錯誤或脆弱寫法，應修正）、Suggestion（風格與可讀性），並引用檔名與行號、給出最小具體修法。這是專業 code review 的格式。

## 第三層：人工逐塊驗收

第三層是人工把關。在實作階段，採用 Claude Code 的 manual approve 模式——AI 每產生一個程式碼修改（diff），都必須經過人工審查後才放行，**而非讓 AI 連續自動套用所有修改**。對於記憶體最複雜的部分（如 PhotoMosaic 載入上千張小圖的生命週期），更是逐個程式碼區塊（block）地審查：AI 寫一塊、人工對照記憶體安全清單驗證一塊、確認無誤後才放行下一塊。

這個人工層的價值在於：它是設計意圖與實作之間的最後一道對齊。subagent 能機械地發現「這裡漏了 delete」，但「這個設計是否符合當初計畫的架構」「這個 friend 是否真的必要」這類判斷，仍需要人工結合對整體設計的理解來把關。

## 成果：流程設計如何促成零洩漏

這套分層流程的成效，直接反映在本專案的記憶體安全結果上——從最簡單的單張影像處理，到 PhotoMosaic 載入上千張小圖、累計超過一千五百萬次（15,530,563）記憶體配置的場景，valgrind 全程回報配置次數與釋放次數完全相等、零洩漏、零錯誤。

更重要的是，這個結果並非偶然，而是流程設計的必然產物。一個具體的例子可以說明：在實作 Sobel 濾鏡時，濾鏡的演算法本體正確地配置了新緩衝區並回傳，但在雙分派的 `RunOn` 接線處，一度漏接了 `replace_buffer`——這會導致回傳的新緩衝區無人接管而整塊洩漏，同時濾鏡也不會生效。由於採用逐塊人工驗收，這個漏接在程式碼放行**之前**就被當場發現並修正，而非等到最終 valgrind 才暴露成一個難以定位的洩漏。

這正是整套流程的核心價值所在：**透過 plan mode 的架構對齊、subagent 的自動化把關、與人工逐塊驗收的三層設計，將記憶體錯誤攔截在它造成損害之前。** AI 提供了生產力，而流程設計守住了品質底線——兩者結合，才是這個對記憶體安全要求極高的專案能全程維持 valgrind 零洩漏的真正原因。
